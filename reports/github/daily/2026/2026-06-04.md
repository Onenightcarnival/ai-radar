## 按主题归类的观察

### Agent 成本控制继续升温

`headroom` 的单日新增继续放大，说明开发者已经开始把 agent 运行成本、工具输出噪声和 RAG chunk 质量当成独立工程问题。它与近期的代码图谱、文档解析、长期记忆项目不同，关注点更靠近每次调用前后的上下文治理。

### Agent harness 和入口仍在拆层

`ECC`、`cc-switch`、`opencode`、`hermes-agent`、`hermes-webui` 同时高位，说明开发者不只在选模型，也在补规则、记忆、入口切换、执行确认和多端访问。真正值得试点的是权限透明、日志可审计、配置可回滚和低权限环境下的真实任务完成质量。

### 文档与网页输入仍是 RAG 上游瓶颈

`markitdown`、`opendataloader-pdf`、`Scrapling` 分别覆盖文件转换、PDF 解析和网页抓取。团队做 RAG 或 agent 试点时，应把格式保真、OCR、权限过滤、失败样本、去重、增量索引和质量评估放进同一条管线，而不是孤立比较单个工具。

### 长期记忆和多端体验需要先定边界

`supermemory`、`hermes-agent`、`hermes-webui` 的组合说明长期上下文和 Web / 移动入口仍有吸引力。采用前要明确哪些内容可以保存、如何删除、谁能查看、工具调用是否需要二次确认，以及移动端误操作如何回放和撤回。

### 安全项目同时覆盖防守和进攻

`trivy` 仍在主榜，`hexstrike-ai` 则把 MCP 和安全工具调用接到 agent。前者适合继续纳入容器、Kubernetes、SBOM、密钥和供应链扫描；后者必须限定授权范围、审计链路和人工复核，不能把自动化 pentest 当成无边界 agent demo。

## 可行动建议

### 给开发者

- 用同一批真实任务对比 `headroom`、普通上下文压缩、代码图谱和手工筛选，记录 token 节省、答案偏差、遗漏证据、延迟和最终 patch 质量。
- 对 `markitdown`、`opendataloader-pdf`、`Scrapling` 做端到端输入测试，重点看复杂文档、动态网页、权限过滤、失败回退、去重和下游索引质量。
- 试用 `ECC`、`cc-switch`、`opencode`、`hermes-agent` 时先限制权限，保留命令执行、配置变化、记忆写入、工具调用和人工确认日志。

### 给技术负责人

- 把 agent 工具链拆成上下文压缩、harness、入口切换、文档 / 网页输入、长期记忆、安全扫描和执行审计几层分别评测，不要按 stars 一次性引入。
- 为外部 agent 工具和 MCP server 建立准入要求：来源验证、版本锁定、权限声明、凭据隔离、日志留存、撤回机制和责任人都要明确。
- 对 pentest agent、长期记忆、移动端 agent 入口和语音 / 视频生成项目设置更高合规门槛，先定义授权范围和人工复核点，再做技术试点。

### 给产品 / 投资观察者

- Agent 周边机会继续分层：上下文压缩、harness、入口切换、长期记忆、文档解析、网页抓取、安全审计和数据库 MCP 都在形成独立卖点。
- 企业更可能为成本可控、权限透明、可审计执行、真实系统集成和质量可评估付费，而不是为单个插件或单次 demo 付费。
- 内容生成和 agent 视频工具仍值得观察，但商业判断要回到授权、成本、审核、稳定性和平台适配，不宜只按单日 stars 判断需求强度。
