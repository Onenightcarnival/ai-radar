## 按主题归类的观察

### Agent 插件和 skills 正在合流

`taste-skill`、`compound-engineering-plugin`、`cursor/plugins`、`claude-code`、`cc-switch` 指向同一条线：开发者不再只比较某个 agent 能不能写代码，而是在补插件规范、技能分发、入口切换和团队流程。真正值得跟进的是这些资产能否版本化、审计、回滚，并被映射到团队自己的测试、设计、文档和发布规则。

### 代码上下文仍是 coding agent 的核心瓶颈

`Understand-Anything`、`codegraph`、`graphify` 等项目连续出现，说明大代码库上下文仍没有被单一 IDE 或 CLI 完全解决。接下来应按真实 issue 对比普通搜索、IDE 索引和图谱索引，记录定位时间、误读文件数、token 消耗、索引耗时和最终 patch 质量。

### 文档入口从格式转换走向解析质量竞争

`markitdown` 继续高位，`liteparse` 新进入热点，说明文档到 Markdown、OCR、PDF 解析和结构化抽取仍是 RAG / agent 落地的前置问题。团队不应只比较支持格式数量，而要用真实文档验证结构保真、表格处理、图片文字、错误回退和权限过滤。

### 业务系统开始重新包装 AI 工作流

`twenty` 把开源 CRM 和 AI 绑定，`claude-code`、`compound-engineering-plugin`、`cursor/plugins` 则把 agent 能力嵌入开发工作台。这个方向的价值不在“AI 标签”，而在能否减少重复录入、自动补上下文、保持权限边界并让人类可以审查每一步。

### 高风险能力需要先定边界

短视频生成、语音克隆、浏览器自动化、长期记忆和 agent 插件都有明确的误用空间。采用前要先确定素材授权、身份冒用防护、凭据隔离、工具调用确认、日志留存和删除机制，再讨论自动化覆盖率。

## 可行动建议

### 给开发者

- 用同一批真实 issue 对比 `Understand-Anything`、`codegraph`、普通搜索和 IDE 索引，记录索引耗时、定位时间、误读文件、token 消耗、测试结果和 patch review 质量。
- 试用 `taste-skill`、`compound-engineering-plugin`、`cursor/plugins` 时，先把外部规则改写成本仓库自己的规范，并记录版本、权限、适用范围和回滚方式。
- 评估 `markitdown`、`liteparse` 时，用真实业务文档跑端到端测试，重点看复杂 PDF、表格、图片文字、增量处理和失败样本。

### 给技术负责人

- 把 agent 工具链拆成代码上下文、插件 / skills、文档入口、记忆、桌面 / 终端入口、安全测试和审计几层分别评测，不要按 stars 一次性引入。
- 为外部插件和 skills 建立准入流程：来源验证、版本锁定、权限声明、凭据隔离、日志留存、撤回机制和责任人都要明确。
- 对 CRM、文档、内容生成和长期记忆类项目先做低权限试点，验证数据权限、人工确认点、删除机制和合规边界。

### 给产品 / 投资观察者

- Agent 周边机会继续分层：插件规范、skills 分发、代码图谱、文档解析、桌面入口、长期记忆和业务系统嵌入都在形成独立卖点。
- 企业更可能为可验证的流程质量、权限可控、审计证据、成本稳定和真实系统集成付费，而不是为单个提示词或一次性 demo 付费。
- 内容生成仍容易带来流量，但商业判断要回到授权、品牌控制、批量稳定性、平台规则和人工审核成本。
