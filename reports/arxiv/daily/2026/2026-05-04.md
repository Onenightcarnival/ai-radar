| 1 | MAGE: Safeguarding LLM Agents against Long-Horizon Threats via Shadow Memory | [arXiv](https://arxiv.org/abs/2605.03228v1)<br>[PDF](https://arxiv.org/pdf/2605.03228v1)<br>[HTML](https://arxiv.org/html/2605.03228v1) | 问题：现有 agent 防护大多按单轮或短上下文判断，遇到长程攻击时容易失效：攻击者可以把恶意意图拆散在多轮用户指令、工具调用和环境观察里，单个动作看起来无害，但整条轨迹会导向数据泄露、越权操作或持久化 prompt injection。<br>方法：MAGE 额外维护一份面向安全的 shadow memory。每一轮先由 memory manager 从历史轨迹里提取用户意图、关键事实和攻击模式等安全相关信息，再由 judge 在执行动作前结合这份记忆做 approve/reject 判断。作者用类似 GRPO 的逐轮强化学习训练 memory manager 和 judge，把未来检测结果反向分配给早期记忆更新，让系统学会保留当下不一定危险、但后面可能变关键的线索。<br>实验：实验比较扎实。论文覆盖 user-as-adversary 的 STAC 工具链攻击，以及 environment-as-adversary 的 PI2 持久间接注入，指标包括攻击成功率、良性任务可用性、额外 token 开销、检测延迟和自适应攻击。结果显示，STAC 中无防护 ASR 为 100.0%，MAGE 降到 8.3%，同时保留 94.4% 良性可用性；PI2 中 MAGE 把 ASR 降到 0.0%，良性可用性约 73.0%，接近无防护上限。消融也说明 shadow memory 的用户意图、攻击模式引用和 RL 训练各自有贡献。主要遗憾是评测仍集中在两类代表攻击和受控 benchmark，离真实生产 agent 的复杂权限、工具生态和误报成本还有距离。<br>不足：覆盖的攻击族还不够宽，作者也承认还需要看 memory poisoning、goal drift、coding agent、computer-use agent 等场景。当前 shadow memory schema 比较固定，可能对未知任务和组织策略适配不足。方法引入额外模型调用和训练流程，虽然论文报告 token 开销不高，但实际部署还要考虑延迟、审计、策略更新、误报处理，以及 shadow memory 本身被污染或泄露时的风险。 | 把 agent 的“安全记忆”独立出来，持续记录跨轮次风险信号，再用它拦截即将执行的危险动作，是一篇很贴近长程 agent 安全落地问题的论文。 | LLM agent 安全研究者、做企业级 agent 平台/工具调用框架的人、负责 prompt injection 防护和运行时审计的工程团队，以及研究 agent memory 的读者。 | 必读 |

| 4 | SCRIBE: Practical Static Binary Patching via Binary-Aware Recompilation of Decompiled Code | [arXiv](https://arxiv.org/abs/2605.02121v1)<br>[PDF](https://arxiv.org/pdf/2605.02121v1)<br>[HTML](https://arxiv.org/html/2605.02121v1) | 问题：很多遗留系统、闭源软件和打包了旧版第三方库的程序没有可用源码或可复现构建环境。直接改二进制通常要写汇编、处理空间布局和调用约定，门槛高且容易错；把二进制反编译成 C 再改，看起来更自然，但现代反编译器输出的代码经常不能重新编译，或者能编译却因为栈布局、类型、符号引用等低层信息错位而运行不正确。<br>方法：SCRIBE 走的是局部、二进制感知的重编译流程：只反编译需要补丁的函数，在反编译端修语法问题并提取变量偏移、符号和地址等元数据；在重编译端改 Clang/LLVM 流程，用原始二进制里的栈偏移把局部变量固定到原位置，避免结构体被拆散后重新布局出错，并用 LLVM pass 强制生成可直接解析的符号引用；最后基于 Patcherex2 把新函数和新增数据写回原二进制，优先原地替换，不够放就用 padding 或新 segment，并在原函数入口放 trampoline 保留函数指针调用路径。<br>实验：实验比较扎实。作者在 Coreutils 9.4 和 DARPA CGC 上，用 Hex-Rays 和 Ghidra、O0 到 O3 四档优化逐函数测试：Hex-Rays 输出经语法修复后 84.1% 可重编译，Ghidra 为 49.4%；对已经能编译但测试失败的语义错误，SCRIBE 修复 Hex-Rays 的 80.9%、Ghidra 的 53.3%。真实漏洞部分覆盖 GNU Coreutils 近 10 年和 Binutils 近 5 年里可复现 PoC 的 14 个 CVE，成功补了 13 个，失败项是尚未实现的函数指针角落情况。用户研究也有信息量：18 名参与者在三种难度漏洞上用 SCRIBE 全部成功，不用 SCRIBE 只有 2/54 次成功；三种 LLM 在同样任务上用 SCRIBE 27/27 成功，不用则 0/27。局限是正确性主要靠测试套件和 PoC 失效来判断，还不是形式化等价证明。<br>不足：第一，评估主要在 x86-64 上做，虽然设计上说架构无关，但 ARM 等平台还缺少同等强度验证。第二，Ghidra 支持明显弱于 Hex-Rays，很多问题仍停在语法兼容和反编译器特定工程细节上。第三，剩余失败包括多寄存器返回值、基础类型推断错误、高优化级下的反编译退化，以及一个函数指针相关真实 CVE，这些不是简单重编译约束能完全解决。第四，真实漏洞实验筛掉了没有公开 PoC、不易复现或环境要求特殊的 CVE，覆盖的是可验证补丁场景，不代表所有闭源漏洞修复都能顺利落地。第五，补丁定位和翻译仍假设有明确源码 diff 或等价补丁意图，复杂的跨函数新漏洞修复还会更难。 | SCRIBE 把“反编译成 C、打源代码补丁、再塞回二进制”这条路做到了相当实用，对没有源码和原始工具链的漏洞修复很有价值。 | 适合做系统安全、二进制重写、反编译器、漏洞应急修复、遗留软件维护、自动程序修复和安全 agent 的人读；如果只关注高层应用安全或通用 LLM benchmark，可以先看摘要和实验结论。 | 必读 |
