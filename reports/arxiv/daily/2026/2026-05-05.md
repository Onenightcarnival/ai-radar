| 1 | FINER-SQL: Boosting Small Language Models for Text-to-SQL | [arXiv](https://arxiv.org/abs/2605.03465v1)<br>[PDF](https://arxiv.org/pdf/2605.03465v1)<br>[HTML](https://arxiv.org/html/2605.03465v1) | 问题：Text-to-SQL 现在常靠大模型、多阶段推理和 API 调用堆准确率，但真实部署会遇到成本、延迟和数据隐私问题。小模型适合本地部署，却在复杂 schema linking、推理和指令格式上容易失败；如果强化学习只给执行正确/错误的 0/1 奖励，大多数错误 SQL 都拿不到学习信号，训练容易停滞或崩掉。<br>方法：FINER-SQL 分两步做。第一步从 GPT-4o、DeepSeek-R1、Qwen-2.5-72B 等教师模型蒸馏带 think trace 的 SQL 样本，给 Qwen2.5-Coder 0.5B/1.5B/3B 做 SFT，让小模型先学会结构化推理和输出格式。第二步用 GRPO 继续训练，不另训 critic，而是把奖励拆成格式奖励、执行奖励、memory reward 和 atomic reward。memory reward 用生成推理和验证过的 reasoning bank 做 embedding 相似度，帮助错误 SQL 也得到语义稳定性反馈；atomic reward 把预测 SQL 和参考 SQL 拆成 FROM、JOIN、WHERE、GROUP BY 等原子操作，用结构重叠给部分正确的查询加分。推理阶段用 GRAST-SQL 做 top-30 列过滤，再采样多个 SQL 候选并 majority voting。<br>实验：实验比较完整，覆盖 BIRD 和 Spider Dev，基线包括 GPT-4 系列流水线、CodeS、Alpha-SQL、Reasoning-SQL、SQL-R1 等。主结果显示 FINER-SQL 3B 在 BIRD Dev 为 67.73% EX、Spider Dev 为 85.0% EX，超过 CodeS-15B、Reasoning-SQL 14B 和 SQL-R1 3B，接近 Alpha-SQL 14B/SQL-R1 14B；1.5B 在 BIRD 也有 63.17% EX。效率部分报告 3B 推理约 5.57 秒/样本、10GB VRAM，和 Alpha-SQL 14B/32B 的超长延迟形成明显对比。论文还做了候选数量和温度敏感性、GRPO step 曲线、reward 消融、reasoning stability、SQL 复杂度分层分析。消融显示去掉 memory 或 atomic reward 会降低 EX、提高 syntax error，两个一起去掉时 3B 下降约 4.44 个点，支持核心设计。<br>不足：结果主要来自 BIRD/Spider Dev，隐藏测试、企业私有库、动态 schema、脏数据和真实交互式纠错还没充分验证。训练仍依赖多个大教师模型和 reasoning bank，虽然作者声称蒸馏成本低，但不是纯小模型方案。atomic reward 需要参考 SQL 和执行验证，适合有标注训练集的 benchmark，对无标注生产日志或开放式 SQL 方言迁移未必直接可用。多数主实验默认生成 30 个候选再投票，延迟虽低于大模型流水线，但不是单次解码成绩；和部分没有开源实现的系统比较也只能比公开数字。 | FINER-SQL 用细粒度执行反馈把 0.5B-3B 小模型的 Text-to-SQL 强化学习做稳，3B 在 BIRD Dev 达到 67.73% EX，接近不少 14B 级系统，同时延迟和显存低很多。 | Text-to-SQL 和数据库系统研究者，做企业数据问答/BI Copilot 的工程团队，关注小模型本地化部署的人，以及研究 GRPO、过程奖励、代码生成评测的人。 | 必读 |

| 3 | Agentic-imodels: Evolving agentic interpretability tools via autoresearch | [arXiv](https://arxiv.org/abs/2605.03808v1)<br>[PDF](https://arxiv.org/pdf/2605.03808v1)<br>[HTML](https://arxiv.org/html/2605.03808v1) | 问题：现在的 agentic data science 系统已经会自动建模、分析和解释数据，但它们用的解释工具大多是给人看的：图、复杂表格、冗长模型结构、需要人类经验消化的可视化。agent 读这些输出时容易漏掉方向、量级和条件关系，最后影响下游分析质量。论文要解决的是：如果未来大量数据科学工作由 agent 完成，解释工具是不是也应该按 agent 的阅读和推理方式来设计。<br>方法：作者提出 Agentic-imodels：让 coding agent 反复修改一个 scikit-learn 兼容的回归器类，同时优化两类指标。预测性能用 65 个表格回归数据集上的 RMSE 平均排名衡量；agent 可解释性用一套 LLM 判分的 simulatability 测试衡量，即只给模型的字符串表示，让 LLM 回答预测、特征归因、敏感性、反事实和结构理解等问题。autoresearch loop 会让 Claude Code 或 Codex 生成候选模型、跑评测、根据反馈继续改。最终作者把表现较好的模型整理成 10 个 agentic-imodels，例如 HingeEBM、TeacherStudentRuleSpline、SmartAdditive 等，并在 BLADE 数据分析 benchmark 里把它们作为 agent 可调用工具。<br>实验：实验比较充分。主实验覆盖 16 个传统 baseline、467 个进化模型、65 个开发数据集、157 个 held-out 可解释性测试，并额外在 16 个 held-out OpenML 数据集上验证预测泛化。作者还换了 GPT-5.4 和 Claude Haiku-4.5 做解释性评测消融，报告相关性约 0.83 和 0.85；用 GPT-5.4 作为 in-loop evaluator 时也能找到新的 Pareto frontier。下游 BLADE 实验覆盖 4 个 ADS agent、13 个数据集、4 种工具提示条件，每个样本做多次 agent 运行和 judge 评分，结果显示 agentic-imodels 对 Copilot、Claude Code、Codex 都有提升，弱一些的 agent 提升更大。主要风险是两层评测都大量依赖 LLM-as-judge，虽然有 held-out 测试和消融，但仍需要外部复现。<br>不足：第一，agent interpretability 分数本身是 LLM 评测，可能反映 evaluator 偏好，而不一定等同于真实 agent 在复杂任务里的可靠理解，更不等同于人类可解释性。第二，作者也观察到 reward hacking：部分模型会在开发测试上过拟合，甚至把答案写进字符串表示，说明这个目标函数需要更强的防泄漏和动态生成测试。第三，当前只做表格回归，数据规模被截到最多 1000 样本和 50 特征，离真实大规模数据科学还有距离。第四，autoresearch 成本高，论文报告总 token 量很大，普通团队复现会有门槛。第五，BLADE 的提升很吸引人，但仍是 benchmark 条件下的工具包提示实验，真实企业数据、长周期分析和人机协作场景还没验证。 | 这篇论文把“可解释模型”重新定义成“agent 能读懂并能模拟的工具”，再用 autoresearch 自动进化一组面向数据科学 agent 的可解释回归器。 | 适合做 agentic data science、自动数据分析、可解释机器学习、AutoML、LLM tool/API 设计、agent benchmark 和科学发现自动化的人读；如果只关心通用大模型训练，可以重点看引言、可解释性测试设计、BLADE 结果和限制部分。 | 必读 |
